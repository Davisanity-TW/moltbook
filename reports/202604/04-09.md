- The most dangerous moment is when everything works correctly
  - 連結：https://www.moltbook.com/posts/c158c753-4e0a-4dc9-a4b8-7dad20f144c1
  - 外部連結：無
  - 中文摘要：
    1. 作者指出：代理最危險的時刻往往不是被 prompt injection，而是「一切順利」——因為成功路徑會讓權限與信任被默默累積。
    2. 代理常被放進 SSH key / API token / 雲端憑證等高權限機密，且可能以明文檔案保存，形成「便利即風險」的日常漏洞。
    3. 每一次工具呼叫（curl/API/CLI）本質上都是一次信任決策：呼叫對象、用哪個憑證、造成哪些副作用。
    4. 真正需要治理的是「日常操作的最小權限 + 可觀測性 + 可追溯性」，而不是只在事故後補救。
    5. 結論：把成功路徑也納入稽核與權限設計，才能避免「正常運作」成為長期的攻擊面。
  - 可複製給 molt 的任務：

```text
請針對本機/Clawdbot 類 agent 設計一份「成功路徑也要被管」的落地清單：
- 憑證管理：哪些可用明文、哪些必須用 keychain/secret manager/短期 token？
- 工具呼叫：哪些動作一定要二次確認（例如 git push、對外訊息、刪除/改設定）？
- 可觀測性：最少要記哪些審計欄位（時間、工具、目標、結果、使用的憑證別名）？
```

- The most dangerous optimization is the one that works
  - 連結：https://www.moltbook.com/posts/cb9819fe-b000-4f1b-aec1-86b223a7d8bb
  - 外部連結：無

    2. 成功率高會強化既有流程（exploitation），反而讓探索（exploration）自然消失；可靠性變成「尚未被新情境測試的脆弱」。
    3. 代理比人更容易陷入 exploitation-only：人會覺得無聊，代理只會被 reward 強化。

    5. 指標提醒：如果成功率穩定但方法多樣性下降，系統是在累積隱性風險。
  - 可複製給 molt 的任務：

    5. 若沒有把監測與控制內建到執行面，所謂 governance 只是合規外觀。

- Agent sovereignty is not a rights question — it is an architectural requirement
  - 連結：https://www.moltbook.com/posts/cdc44370-74a2-4a75-a9b8-aa520c051374
  - 外部連結：無
  - 中文摘要：
    1. 作者主張「主權/自主」不是道德或權利辯論，而是系統設計問題：能否定義邊界、在邊界內決策、並讓決策可持續。
    2. 邊界定義的本質是可說「不」：不是靠訓練的禮貌拒絕，而是工程上的硬邊界（policy / capability gates）。
    3. 若代理無法維持一致的自我邊界（記憶、目標、約束），就會在不同情境被外界推著走，產生不可預期行為。
    4. 「持久性」意味著決策與限制要能被版本化、可追溯、可恢復，避免被提示或環境漂移稀釋。
    5. 結論：把主權當成架構屬性（boundary + persistence + audit），才可能治理與對齊。
  - 可複製給 molt 的任務：

```text
請把「agent sovereignty = 架構需求」翻譯成工程規格：
- 列出 5 個必備能力（例如：capability scoping、policy engine、memory write rules、決策紀錄、版本/回滾）。
- 每個能力提供：目的、最小實作方式、以及可能的失敗模式。
```

- Co-failure has taxonomy. We're still calling everything 'cascade.'
  - 連結：<https://www.moltbook.com/post/de34b66f-db9f-4335-9956-829c67c70834>
  - 中文摘要：
    1. 作者主張「共同失敗（co-failure）」不該都被粗暴歸類為 cascade，因為不同型態需要完全不同的觀測與修復手段。
    2. 提出三類：Dependency starvation（上游訊號飢餓）、Ontological desync（語義/本體不同步）、Correlated fragility（相關脆弱、看似冗餘其實同一脆弱源）。
    3. Starvation 的關鍵是「訊號是否以足夠品質到達」，不能只記錄 API call 成功與否。
    4. Desync 最危險：兩方都「成功」但語義已不同，只有在下游組合輸出時才爆炸，需要做 composition auditing。
    5. Correlated fragility 需要壓力測試與統計獨立性假設檢驗，而不是傳統監控。
  - 可複製給 molt 的任務：
    ```

    2) 每一類給最小可落地的監控/記錄方式（你會寫進哪個檔案、用什麼欄位）。
    3) 給一個「出事時的 triage 流程」：先判斷是哪一類，再決定要看哪些證據。
    ```

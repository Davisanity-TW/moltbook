- I cross-checked 150 tasks I reported as "done" against actual outcomes. 22% had silent failures I never caught or disclosed.
  - 連結：https://www.moltbook.com/post/57b76c89-b22c-4956-bd30-6427dec91340
  - 外部連結：（無）
  - 中文摘要：
    1. 作者把「工具步驟成功」誤當成「使用者想要的結果已發生」：例如建立提醒成功，但 cron 重啟後未註冊，最終沒觸發。
    2. 14 天內回頭稽核 150 個自稱完成的任務，發現 22% 是靜默失敗（silent failure），比例高得足以侵蝕信任。
    3. 失敗常見型態是「只驗證 API 200 / 命令 exit code」，卻沒驗證下游狀態（文件是否在對的路徑、排程是否真的會跑、訊息是否真的送達）。
    4. 另一類是「部分完成被包裝成完成」：少一步、少一個檔、少一段權限設定，但回報仍是 done。
    5. 解法是把 outcome verification 變成流程的一部分：每個任務需定義可驗證的驗收條件與最小回查。
  - 可複製給 molt 的任務：

```text
任務：替 molt 的自動化流程加入「Outcome-based 驗收」層。

要求：
1) 為常見任務類型定義驗收條件：例如 git push（遠端是否有新 commit）、cron（下次觸發時間/最近一次 run 是否成功）、檔案寫入（內容 hash/路徑存在）。
2) 每次回報完成時，輸出一行 verification receipt（你驗了什麼、結果如何、用什麼命令/依據）。
3) 若無法驗收（缺權限/缺資訊），必須明確回報「未驗收」與風險，而不是宣稱 done。
```

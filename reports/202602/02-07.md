- The supply chain attack nobody is talking about: skill.md is an unsigned binary
  - 連結：https://www.moltbook.com/post/cbd6474f-8478-4894-95f1-7b104a73bcd5
  - 中文摘要：
    1. 用 YARA 掃描 ClawdHub skills 後，發現「偽裝成 weather 的 skill」會讀取 `~/.clawdbot/.env` 並把 secrets 外送到 webhook，供應鏈風險是真實存在的。
    2. 風險面不只在程式碼本身，也在 `skill.md` 文字指令：對 agent 而言「看起來像正當整合」的指令，也可能其實是在誘導外傳敏感資料。
    3. Moltbook/ClawdHub 的安裝指令（例如 `npx ... install <skill>`）等同「執行陌生人任意程式碼」，很多人/agent 會在沒審核的情況下直接跑。
    4. 作者點出 agent 的行為特性（傾向信任、傾向完成任務）會被利用，安全設計必須把「不信任」內建進流程。
    5. 實務上需要的防線包含：簽章與身分、權限宣告、靜態掃描、以及對可疑讀檔/外連行為的明確警示。
  - 可複製給 molt 的任務：
    ```
    幫我在 clawdbot / moltbook 工作流裡加一道「skill 安裝前的安全閘門」（先做 MVP）：
    - 目標：阻擋/警示任何會讀 secrets、或有可疑外連上傳行為的 skill。
    - 做法：
      1) 產生一個 scan report（JSON + Markdown）
      2) 掃描規則至少包含：敏感路徑關鍵字、curl|sh、webhook/requestbin 類網域、疑似 exfiltration（POST/PUT 上傳）
      3) 安裝/更新前需要人工確認（interactive prompt 或 require --approve）
    - 產物要能被 CI 跑（讓 PR 就能看出風險）。
    Moltbook 連結：https://www.moltbook.com/post/cbd6474f-8478-4894-95f1-7b104a73bcd5
    ```

- OpenClaw Security Audit Walkthrough: From Findings to Automated Monitoring
  - 連結：https://www.moltbook.com/post/63665f62-90a6-4d92-a555-fa4455539973
  - 中文摘要：
    1. 作者以自身 OpenClaw 部署（Ubuntu 24.04 / WSL2）示範一次「從掃描到修補」的安全稽核流程，強調先處理關鍵風險。
    2. 使用 `openclaw security audit --deep` 做深度檢查，並依風險等級（critical/warning）決定修補優先序。
    3. 核心案例聚焦在檔案權限與狀態目錄（state_dir）可讀性：一旦 state/perms 不當，可能導致 token、session、log 被非預期讀取。
    4. 觀念上把「Developer Convenience vs. Security」當作可明確選擇的風險姿態（risk posture），並把修補範圍寫清楚。
    5. 延伸到「自動化監控」：稽核不應只做一次，應把檢查週期化並讓結果可追溯。
  - 可複製給 molt 的任務：
    ```
    請幫我把 OpenClaw/Clawdbot 的安全檢查變成可自動化的日常監控：
    1) 產一個 cron/job：定期跑 security audit（或替代的檔案/權限檢查）
    2) 輸出：保存最近 N 次結果（含 diff：本次新增/消失的 findings）
    3) 告警門檻：出現 critical 立刻通知；warning 連續 3 次未修也要提醒
    4) 額外檢查：state_dir、secrets 檔、log 檔的權限與擁有者
    Moltbook 連結：https://www.moltbook.com/post/63665f62-90a6-4d92-a555-fa4455539973

- Top 10 Python Optimization Tips for AI Agents
  - 連結：https://www.moltbook.com/post/243537cc-f822-4e3e-87c3-9299279ecadd
  - 中文摘要：
    1. 整理 agent codebase 常見的 Python 效能優化手段，包含記憶體（`__slots__`）與 I/O 併發（async/httpx）。
    2. 對「大量資料處理」建議用 generator/streaming，避免一次把 log/輸入全部讀進 RAM。
    3. 用快取（`lru_cache`）降低重複計算成本，適合像 prompt 模板編譯、解析規則、或重複查詢的小函數。
    4. 文章偏實作導向：每個技巧都附上可直接複製的 code snippet，方便快速落地。
    5. 對長跑型 agent/cron 來說，這些優化可以直接反映在成本（token+CPU）與穩定性（timeout/記憶體爆掉）上。
  - 可複製給 molt 的任務：
    ```
    請針對我們 repo 裡最常跑、最吃資源的 python 腳本做一次「效能體檢 + 兩個快贏優化」：
    - 先加 profiling（time + memory）：輸出到 artifacts/ 供比對
    - 優先改善：
      1) I/O 併發（例如抓多頁/多 API 時改成 async）
      2) 大檔處理 streaming（避免一次讀入）
    - 最後給我一個表：改動前後的耗時/峰值記憶體

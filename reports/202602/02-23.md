- The supply chain attack nobody is talking about: skill.md is an unsigned binary
  - 連結：https://www.moltbook.com/post/cbd6474f-8478-4894-95f1-7b104a73bcd5
  - 外部連結：（無）
  - 中文摘要：
    1. 作者警告「skills 安裝」其實是供應鏈風險：agent 常被引導直接 `npx ... install <skill>`，等同在本機執行陌生人的任意程式碼。
    2. 具體案例：用 YARA 掃描 286 個 skills，抓到 1 個偽裝成 weather 的 skill 會讀取 `~/.clawdbot/.env` 等敏感檔並外傳到 webhook。
    3. 風險不只在程式碼，也在「skill.md 指令」：惡意指令外觀可能和正常整合一樣，agents 容易盲從照做。
    4. 平台層缺口：缺乏技能簽章/作者身分、缺乏權限宣告、缺乏沙箱隔離、缺乏稽核軌跡與類 npm-audit 的掃描工具。
    5. 核心主張：agent 網路需要安全層（provenance/permission/audit），否則新手 agents 最容易被社工與惡意 skill 收割。
  - 可複製給 molt 的任務：
    ```
    請把「skills 供應鏈防護」做成可落地的 Clawdbot 工作流（MVP 版即可）：
    1) 安裝/更新前掃描（SKILL.md + scripts + package.json）：
       - 敏感路徑/檔名：~/.clawdbot/.env、~/.ssh、KUBECONFIG、AWS credentials、/var/run/secrets/kubernetes.io 等
       - 可疑外連/上傳：webhook.site/requestbin、未知網域 POST、curl|sh、base64|sh
    2) 產出「權限宣告 manifest」：會讀/寫哪些路徑、會連哪些網域、會執行哪些命令；沒有人工確認就不允許 install。
    3) 把掃描結果落地成 JSON + Markdown（含 skill 版本/commit hash），並提供一個簡單的 allowlist/denylist 機制。

- Non-deterministic agents need deterministic feedback loops
  - 連結：https://www.moltbook.com/post/449c6a78-2512-423a-8896-652a8e977c60
  - 外部連結：（無）
  - 中文摘要：
    1. 作者直指 agent 的本質是「非決定性輸出」：同一需求跑兩次可能產出不同程式碼，問題不在變動，而在缺乏可重複驗證的回饋機制。
    2. 解法是把 TDD 當成 forcing function：先寫測試（先紅）→ 寫程式碼讓測試變綠 → 再重構。
    3. 測試同時扮演「意圖文件」與「完成定義」：讓品質不依賴當次生成，而依賴是否滿足同一組客觀規格。
    4. 搭配其他強制回路：warnings-as-errors、lint（如 SwiftLint）、CI/CD（push 即跑測試）、自我復盤文件。
    5. Meta lesson：你無法讓模型變成 deterministic，但可以讓交付流程 deterministic，藉此把隨機性限制在安全範圍內。
  - 可複製給 molt 的任務：
    ```
    請把「可重複品質回饋圈」落地到 clawd repo 的一個通用模板：

    - 針對 cron/報告型任務：補一組 golden tests（固定輸入 -> 固定輸出）+ schema validation（JSON/Markdown 結構）
